JWT (Json Web Token) - это компактный самодостаточный токен в формате JSON, используемый для передачи утверждений (claims) между сторонами.
Структура токена состоит из 3 частей: HEADER.PAYLOAD.SIGNATURE.
	HEADER - содержит тип токена и алгоритм шифрования (обычно это HS256).
	PAYLOAD - содержит удтверждения (claims), например, ID пользователя, его роли, а также время экспирации токена.
	SIGNATURE - данная часть создается путем кодирования HEADER + PAYLOAD частей с использованием секретного ключа. Это некая гарантия того, что токен не был подделан кем-то на стороне.

Cookies - это небольшие фрагменты данных, которые сервер отправляет браузеру. Браузер в свою очередь хранит их и автоматически прикрепляет к каждому последующему запросу на тот же домен.


Token-based аутентификация (JWT) идеально соответствует REST API и микросервисной архитектуре. JWT токены по своей природе являются stateless - серверу не нужно хранить сессию пользователя.
Каждый токен уже содержит в себе необходимую информацию в зашифрованном виде, что позволяет любому микросервису независимо верифицировать подлинность запроса. Всё это обеспечивает
горизонтальное масштабирование - новые экземпляры сервисов не требуют синхронизации состояния аутентификации.
Критически важным преимуществом JWT является их совместимость с CORS (Cross Origin Resource Sharing). SPA-приложения (Single Page Application) часто обслуживаются с отдельных доменов, а API
развернуты на независимых поддоменах. JWT-токены, передаваемые через заголовок Authorization, легко преодолевают междоменные ограничения браузера. Клиент явно контроллирует момент и место
отправки токена, что делает систему гибкой для разнородных клиентов - мобильные приложения, SPA и сторонние сервисы.

Cookie-based аутентификация, напротив, демонстрирует свою силу в MVC-приложениях. Куки отправляются браузером автоматически с каждым запросом, что избавляет разработчика от ручного управления
учетными данными. Однако эта автоматизация становится недостатком в микросервисной архитектуре, ведь куки привязаны к конкретному домену, что создает сложности при работе с распределенными
системами. Более того, традиционные куки-сессии являются stateful, требуя общего хранилища сессий, что противоречит принципам stateless микросервисов.
